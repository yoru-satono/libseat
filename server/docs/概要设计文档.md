## 1. 系统概述

### 1.1 系统目标

LibSeat 后端是一套面向高校图书馆的座位预约管理系统，以 RESTful API 的形式为前端 SPA 和潜在的移动端提供服务。核心能力：

- 实时查询座位可用状态（含时间段维度过滤）
- 完整的预约生命周期管理：`ACTIVE → CHECKED_IN → IN_USE → COMPLETED`，以及 `CANCELLED` / `NO_SHOW` 异常路径
- 等待队列：座位释放后自动通知队列中的下一位用户
- 多级规则：全局规则 + 图书馆级覆盖规则，灵活控制开放时间、预约时长限制、爽约阈值等
- 管理员后台：用户管理、预约管控、信息修改申请审核、审计日志

### 1.2 系统边界

```
                 ┌──────────────────────────────┐
  前端 SPA / App │                              │ PostgreSQL 17
  ───────────────►  libseat-server (Spring Boot) ◄─────────────
  HTTP REST API  │                              │
                 │  SMTP 邮件服务（激活/重置密码）│
                 └──────────────────────────────┘
```

系统仅提供 HTTP API，不包含 WebSocket 或消息队列；通知功能通过站内通知 + 前端轮询实现。

## 4. 模块划分

### 4.1 认证模块（AuthService + AuthController）

负责用户身份注册、邮箱激活、登录、Token 刷新、密码找回、邮箱变更确认。

核心流程：
- 注册 → 发送激活邮件（`email_tokens`）→ 激活账号（`status: INACTIVE → ACTIVE`）
- 登录 → 失败计数 / 锁定逻辑 → 签发 JWT 双 Token
- 刷新：验证 refreshToken 签发新对 Token

### 4.2 用户模块（UserService + UserController）

当前登录用户的自助操作：
- 查看/修改个人资料（手机号、院系可直接修改）
- 提交关键字段修改申请（学号/姓名/院系，需管理员审核）
- 导出个人预约记录（xlsx）

### 4.3 座位模块（SeatService + SeatController）

- 带时段过滤的座位列表查询（匿名可访问）：JPA Specifications 动态拼接条件，时段冲突检测排除已被预约的座位
- 座位 CRUD（管理员权限，通过 AdminController 或 SeatController 路由）

### 4.4 预约模块（ReservationService + ReservationController）

预约的完整生命周期：

```
创建(ACTIVE) → 签到(CHECKED_IN) → 开始使用(IN_USE) → 完成(COMPLETED)
                                                    ↗ 取消(CANCELLED)
                                   ↗ 爽约(NO_SHOW，由定时任务标记)
```

关键操作：
- **创建**：验证座位可用、时段不冲突、规则合规（时长、提前天数、日累计）
- **取消**：距开始时间 ≥ 30 分钟可取消；取消后触发等待队列通知
- **签到**：在签到窗口期内（开始前 N 分钟至宽限 N 分钟）凭 QR token 签到
- **续约**：结束前 15 分钟内可续约，新建子预约（`parent_id` 关联）

### 4.5 等待队列模块（WaitlistService + WaitlistController）

- 用户对已被预约的时段加入等待队列
- 座位释放时（取消/爽约），自动通知队列头部用户，设置 24 小时确认窗口
- 窗口过期后将通知权转给队列下一位

### 4.6 通知模块（NotificationService + NotificationController）

站内通知的存取操作：列表（支持已读/未读筛选）、标单条已读、全部标已读、删除、未读数量查询。

通知由各业务模块（预约/等待队列/定时任务）在事件发生时同步写入 `notifications` 表。

### 4.7 管理员模块（AdminService + AdminController）

管理员专属接口，全部挂载在 `/v1/admin/**`，路由层由 `hasRole("ADMIN")` 守护。

| 子功能 | 说明 |
|--------|------|
| 用户列表/详情 | 支持角色/状态筛选 |
| 用户操作 | 锁定、解锁、重置密码，写入审计日志 |
| 预约管理 | 全平台预约查询 + 管理员强制取消 |
| 修改申请审核 | 批准/拒绝用户的关键字段修改申请 |
| 图书馆 CRUD | 新建/编辑/删除图书馆 |
| 系统规则 | 读取/更新全局规则或图书馆级覆盖规则 |
| 审计日志 | 只读，支持多维度筛选 |
| 数据导出 | 管理员维度预约记录 xlsx 导出 |

### 4.8 辅助服务

| 服务 | 职责 |
|------|------|
| JwtService | 签发、验证、提取 Claims，区分 access/refresh token |
| EmailService | 发送激活邮件、密码重置邮件、邮箱变更确认邮件 |
| ExcelExportService | Apache POI 生成预约记录 xlsx，限 5000 条（A0420） |
| LibraryService | 图书馆 CRUD，被 LibraryController（公开只读）和 AdminController（写操作）共用 |

## 5. 数据库设计概要

### 5.1 数据库选型

PostgreSQL 17，利用以下特性：
- **原生 ENUM 类型**：`user_role`、`reservation_status` 等 9 个枚举，比字符串列约束更强
- **UUID 主键**：除 `system_rules`、`audit_logs`（BIGINT IDENTITY）外，所有主表均用 `gen_random_uuid()`
- **JSONB**：`audit_logs.detail` 存储变更前后快照，无需预定义字段
- **触发器**：`set_updated_at()` 函数由 BEFORE UPDATE 触发器自动维护 `updated_at`

### 5.2 核心实体关系

```
libraries ──< seats ──< reservations >── users
                │           │
                └──< waitlists          │
                                        │
users ──< notifications                 │
users ──< user_change_requests          │
users ──< email_tokens                  │
users ──< audit_logs (admin_id) ────────┘

system_rules (library_id IS NULL → 全局; 非NULL → 图书馆级覆盖)
reservations.parent_id → reservations (续约链)
```

### 5.3 核心表概览

| 表名 | 主键类型 | 行数量级 | 说明 |
|------|---------|---------|------|
| `libraries` | UUID | 小（< 100） | 图书馆基础信息 |
| `users` | UUID | 中（万级） | 含账号状态、爽约次数、锁定时间 |
| `email_tokens` | UUID | 中（随注册量增长） | 激活/重置/邮箱变更令牌，定时清理 |
| `seats` | UUID | 中（千级） | 座位基础信息 + 设施属性 |
| `reservations` | UUID | 大（百万级） | 预约主表，含续约链（parent_id） |
| `waitlists` | UUID | 中 | 等待队列，含通知过期时间 |
| `system_rules` | BIGINT | 极小（< 10） | 全局规则 + 各馆覆盖规则 |
| `user_change_requests` | UUID | 小 | 关键字段修改申请，审核流程 |
| `notifications` | UUID | 大 | 站内通知，含已读状态 |
| `audit_logs` | BIGINT | 大（只追加） | 管理员操作审计，JSONB 存快照 |

### 5.4 关键索引

| 表 | 索引字段 | 用途 |
|----|---------|------|
| `reservations` | `(seat_id, date, start_time, end_time)` | 时段冲突检测 |
| `reservations` | `(user_id, date, start_time, end_time)` | 用户当日预约查询 |
| `reservations` | `(status, date)` | 定时任务扫描 |
| `waitlists` | `(seat_id, date, start_time, status, created_at)` | 队列有序通知 |
| `notifications` | `(user_id, is_read, created_at)` | 未读数量 + 列表查询 |
| `audit_logs` | `(created_at)`, `(target_type, target_id)` | 时间/目标维度检索 |
| `seats` | `(library_id, floor, area, status)` | 多条件座位筛选 |

| 枚举名 | 取值 |

| `user_role` | STUDENT / TEACHER / ADMIN |
| `user_status` | INACTIVE / ACTIVE / LOCKED / SUSPENDED |

| `seat_status_t` | AVAILABLE / UNAVAILABLE |

## 6. API 设计规范

### 6.1 基础规则

- **Context Path**：`/api`（`server.servlet.context-path`）
- **版本前缀**：`/v1`（控制器 `@RequestMapping`）
- **完整前缀**：所有接口路径以 `/api/v1` 开头
- **URL 风格**：复数名词 + 连字符，禁止动词（`/reservations/{id}/checkin` 而非 `/checkinReservation`）
- **HTTP 状态码**：统一返回 `200`，业务成功与否由响应体 `code` 字段判断

### 6.2 响应格式

```json
{
  "code": "00000",
  "message": "success",
  "data": { ... },
  "timestamp": "2026-05-11T10:00:00+08:00"
}
```

错误时 `data` 为 `null`，`code` 为对应错误码，`message` 为可读描述。

### 6.3 分页规范

- 请求参数：`page`（从 **1** 开始，默认 1）、`pageSize`（默认 20，最大 100）
- 响应结构：`PageResult<T>` — `{ items, total, page, pageSize, totalPages }`

### 6.4 接口权限矩阵

| 路径 | 方法 | 权限 |
|------|------|------|
| `/v1/auth/**` | ALL | 公开 |
| `GET /v1/seats/**` | GET | 公开（匿名可查询座位） |

## 7. 安全设计

### 7.1 JWT 双 Token 方案

| Token | 有效期 | 签发方式 | 用途 |
|-------|--------|---------|------|
| accessToken | 2 小时（可配置） | 登录/刷新时签发 | 请求头 `Authorization: Bearer <token>` |
| refreshToken | 7 天（可配置） | 仅在登录时签发 | 向 `/auth/refresh` 换新 accessToken |

Token 使用 HS256 算法，密钥由 `app.jwt.secret` 环境变量注入。Claims 中携带 `userId` 和 token 类型（`access` / `refresh`），过滤器层通过类型字段拒绝将 refreshToken 用于业务接口。

### 7.2 认证流程

```
请求到达 JwtAuthenticationFilter
  ├─ 无 Authorization 头 → 继续（匿名访问）
  ├─ 是 refreshToken → 设置 ATTR_JWT_ERROR=TOKEN_INVALID，继续
  ├─ accessToken 过期 → 设置 ATTR_JWT_ERROR=TOKEN_EXPIRED，继续
  ├─ accessToken 无效 → 设置 ATTR_JWT_ERROR=TOKEN_INVALID，继续
  └─ 有效 → 从 DB 查询 User → 设置 SecurityContext Authentication

SecurityConfig.exceptionHandling:
  AuthenticationEntryPoint → 读取 ATTR_JWT_ERROR → 返回 A0201 / A0202 / A0100（Result 格式）
  AccessDeniedHandler → 返回 A0301
```

### 7.3 密码安全

- 存储：BCrypt（Spring Security `BCryptPasswordEncoder`）
- 登录失败计数：连续失败 5 次 → `locked_until = now + 30分钟`（`ACCOUNT_LOCKED`，A0601）
- 重置：邮件令牌有效期 24 小时，使用后标记 `used_at`

### 7.4 账号暂停机制

爽约次数（`no_show_count`）达到 `SystemRules.noShowThreshold`（默认 3 次）时：
- `suspended_until = now + suspendDays`（默认 7 天）
- `no_show_count` 重置为 0
- 发送站内通知（ACCOUNT_SUSPENDED）
- 暂停期间创建预约时返回 `A0602`

### 7.5 CORS 配置

允许 `http://localhost:*`（本地开发），生产环境需通过 `allowedOriginPatterns` 配置正式域名。

## 8. 错误码体系

采用 5 位分层错误码（参考阿里巴巴 Java 开发手册）：

| 前缀 | 含义 | 示例 |
|------|------|------|
| `00000` | 成功 | — |
| `A0xxx` | 用户端错误 | 认证、鉴权、参数、账号状态 |
| `B0xxx` | 业务规则错误 | 时段冲突、超限、窗口期 |
| `C0xxx` | 服务端错误 | 内部异常 |

### 关键错误码

| 码 | 说明 | 客户端处理 |
|----|------|-----------|
| A0100 | 未登录 | 跳转登录页 |
| A0201 | accessToken 过期 | 调 /auth/refresh 刷新 |
| A0202 | refreshToken 无效/过期 | 重新登录 |
| A0301 | 无权限 | 提示无权访问 |
| A0400 | 参数校验失败 | 展示 message 字段 |

## 9. 关键领域规则

### 9.1 座位可用性判断

`GET /v1/seats` 若携带 `date + startTime + endTime`，后端通过 JPA Specifications 排除在该时段有 `ACTIVE / CHECKED_IN / IN_USE` 状态预约的座位，返回实际可预约的座位列表。

### 9.2 预约规则校验（创建时）

1. 座位状态为 `AVAILABLE`
2. 时段不与该座位现有有效预约冲突
3. 预约日期 ≤ 今天 + `advanceDaysMax`（超出返回 B0201）
4. 预约时长在 `[singleMinMinutes, singleMaxHours*60]` 范围内（B0202）
5. 开始/结束时间在图书馆 `[openTimeStart, openTimeEnd]` 内（B0203）
6. 当日累计预约时长 + 本次时长 ≤ `dailyMaxHours * 60`（B0210）

规则取值优先级：**图书馆级覆盖规则 > 全局默认规则**（`library_id IS NULL`）。

### 9.3 取消规则

- 预约状态须为 `ACTIVE`
- 当前时间距预约开始时间 ≥ 30 分钟（B0301）
- 取消成功后触发等待队列通知

### 9.4 续约规则

- 当前状态为 `CHECKED_IN` 或 `IN_USE`
- 当前时间在预约结束前 15 分钟内（B0501）
- 新结束时间在日累计时长限制内
- 续约时段无其他人预约（B0500）
- 新建续约 `Reservation` 记录，`parent_id` 指向当前预约

### 9.5 签到规则

签到时间窗口 = `[开始时间 - checkinEarlyMinutes, 开始时间 + checkinLateMinutes]`（B0400）。  
超过 `checkinLateMinutes` 未签到的 `ACTIVE` 预约由定时任务标记为 `NO_SHOW`。

### 9.6 系统规则分级

```
SELECT * FROM system_rules WHERE library_id = :libraryId  -- 库级规则（优先）
UNION ALL
SELECT * FROM system_rules WHERE library_id IS NULL        -- 全局规则（备选）
LIMIT 1
```

## 10. 定时任务

### 10.1 预约状态维护（每 5 分钟）

`ReservationScheduler.processExpiredReservations()`

1. **爽约标记**：扫描 `status=ACTIVE` 且已过签到宽限期的预约 → 标为 `NO_SHOW`，累加用户爽约次数，超阈值暂停账号，触发等待队列通知
2. **完成标记**：扫描 `status=CHECKED_IN / IN_USE` 且结束时间已过的预约 → 标为 `COMPLETED`

### 10.2 过期令牌清理（每天凌晨 3 点）

`ReservationScheduler.cleanExpiredTokens()`

批量删除 `email_tokens` 中 `expires_at < now` 且未使用的记录，防止表体积持续增长。

## 12. 设计约束

| 约束 | 说明 |
|------|------|
| 无 WebSocket | 通知依赖前端轮询 `/notifications/unread-count`（每 60 秒） |
| 无 Redis | 无会话存储、无缓存层；Token 无服务端状态，靠 JWT 自包含信息 |
| JWT 无状态登出 | `/auth/logout` 为语义接口，实际登出由客户端丢弃 Token 完成 |
| Schema 手动管理 | 禁止 `ddl-auto=create/update`，Schema 变更须手动修改 `01_schema.sql` |
| 导出上限 | Excel 导出最多 5000 条（A0420），防止 OOM |
| Jackson 3.x 包名 | Spring Boot 4.x 中 Jackson 包前缀为 `tools.jackson`，非 `com.fasterxml.jackson` |
| PostgreSQL ENUM 大写 | DDL 中 ENUM 值必须大写，与 Java `enum.name()` 对齐，否则 Hibernate 类型转换报错 |
| `@Modifying` 清缓存 | 批量更新 Repository 方法必须设置 `clearAutomatically = true`，防止一级缓存脏读 |
